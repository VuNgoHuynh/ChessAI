# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

A pygame chess game where a human plays against an MCTS AI. Two start menus let the player pick their side (White or Black; the AI takes the other) and then the game mode (Classic = standard start, or Random = a random Chess960 / Fischer-random position). The board is oriented so the human's pieces always sit at the bottom of the screen.

## Running

There is a bundled virtualenv (`PythonAI3.12/`, Python 3.12.2) with `pygame` and `python-chess` 1.10.0 installed. Always use its interpreter â€” the game depends on relative paths (`Images/`, `SoundEffect/`), so run from the repo root:

```bash
./PythonAI3.12/Scripts/python.exe Main.py
```

There is **no build step**. `Main.py` now exposes a `main(max_frames=None)` function guarded by `if __name__ == "__main__"`, so importing it no longer starts the game loop. `main` accepts an optional `max_frames` cap used by headless tests. Runtime deps are pinned in `requirements.txt` (`pygame==2.5.2`, `chess==1.10.0`); the bundled `PythonAI3.12/` venv is *not* checked in (it's gitignored) â€” CI installs fresh from `requirements.txt`.

Because the game opens a GUI window, it can't be exercised interactively in a headless environment, but the logic *can* be driven headlessly: set `SDL_VIDEODRIVER=dummy` and `SDL_AUDIODRIVER=dummy`, then either exercise the `entity` classes directly or call `Main.main(max_frames=N)` (set `Main.ENGINE_DELAY = 0` to skip the AI think-delay). Run with `PYTHONPATH` pointing at the repo root so the `entity` package resolves.

## Testing & CI

Tests live in `tests/` and run under **pytest** (`./PythonAI3.12/Scripts/python.exe -m pytest`). `pytest.ini` puts the repo root on `sys.path` (so `import entity`/`import Main` resolve) and `conftest.py` sets the SDL dummy drivers **before** pygame is imported, so the whole suite is headless. A `make_board` fixture builds an `entity.Board` on a 1Ã—1 dummy display (empty images â€” `draw()` tolerates missing art) and re-initialises pygame if a prior test tore it down (e.g. `Main.main` calls `pygame.quit()`), keeping tests order-independent. Coverage: board coordinate round-trip (both orientations), `HumanPlayer` click flow, `EnginePlayer` move selection, and a `Main.main` game-loop smoke. **MCTS determinism in tests:** serial (`num_workers=1`) searches `random.seed(...)` immediately before the call to stay reproducible; the parallel path reseeds each worker from fresh OS entropy by design, so its test asserts *structure* (a legal move), not a statistical outcome, and is skipped on Windows (spawn-under-pytest is unsafe there â€” Linux CI covers it via fork).

**Self-healing CI/CD** (`.github/workflows/ci.yml`) runs on push/PR to `master`, using only the built-in `GITHUB_TOKEN` (no secrets). Three jobs: **`test`** installs deps (with a 3Ã— retry loop so transient pip/network failures self-heal) and runs pytest â€” the real gate; **`autofix`** (same-repo PRs only â€” fork tokens are read-only) auto-commits **only import sorting** (`ruff check --fix --select I`) back to the PR branch, then reports remaining lint non-blockingly. It deliberately does *not* auto-apply `F` fixes: deleting an "unused" import could remove a name another in-flight branch is about to use, silently breaking it on merge (import *sorting* only reorders, never drops a name, so it's merge-safe). **`heal`** runs only when `test` fails on a `master` push and opens *or updates* a single `ci-failure`-labelled tracking issue (deduplicated, so repeated red builds don't spam). Lint config is `ruff.toml`, deliberately conservative (`select = ["F", "I"]`, no `ruff format`) so it never churns the hand-wrapped style. Note: a `GITHUB_TOKEN` push does not re-trigger workflows, so the autofix commit does not start a new run (the `[skip ci]` on it is just belt-and-braces).

## Architecture

The game runs on **python-chess** for all rules and board state. The `entity/` package is the whole implementation, driven by `Main.py`. (An older hand-rolled, pixel-keyed prototype â€” `Main_legacy.py`, which depended on a `ChessEngine.py` â€” has been removed; the `entity/` architecture fully supersedes it.)

### `entity/` package (live implementation)

- **`Board`** (`entity/board.py`) â€” wraps a `chess.Board` and owns all pygame rendering. Converts between pixels and squares (`square_at(pos)`, `pixel_of(square)`), lists legal destinations (`target_squares(square)`), and applies moves with the right sound (`push(move)` uses `SoundEffect.classify` to pick Castle/Capture/Promote/Move *before* `board.push`, then upgrades to Check by inspecting check state *after*). Holds a `SoundEffect` in `self.sound`. `draw()` renders squares, highlights, pieces, and â€” once `board.is_game_over()` â€” a **Game Over** overlay. Holds transient UI state in `self.selected` and remembers the `last_move` for the move trace.
  - **Highlights** (`_draw_highlights`): the last move's from/to squares are tinted with a semi-transparent yellow surface (`TRACECOLOR`/`TRACEALPHA`, alpha-blended so the square shows through) â€” this is the "trace" of the move, set in `push()` and drawn first so an active selection paints over it. Then the solid selection square, its legal-target squares, and the king-in-check square.
  - **Game Over** (`_draw_game_over`): dims the board with a translucent black veil and centres "Game Over" plus a subtitle from `board.outcome()` (e.g. "Checkmate â€” Black wins", "Stalemate â€” Draw"), and draws a **"New Game" button** (rect stored in `self.new_game_rect`). Fonts are built lazily once via `_get_fonts()` and guarded in try/except so headless runs (no font subsystem) just draw the veil â€” but the button rect is drawn/stored *before* the no-font early return so it stays clickable headless. `new_game_clicked(pos)` returns True when the game is over and `pos` hits the stored rect; `Main.py` handles the actual restart.
- **`HumanPlayer`** (`entity/human_player.py`) â€” turns mouse clicks into moves. Two-click flow: first click selects one of the player's own pieces, second click picks the destination; `handle_click(pos, board)` returns a `chess.Move` when a legal move is formed (else `None`) and drives `board.selected`. Promotions are auto-queened in `_build_move`.
- **`EnginePlayer`** (`entity/engine_player.py`) â€” the AI, which plays whichever colour it is constructed with (the side the human didn't pick). A **Monte Carlo Tree Search** engine: `EnginePlayer(color, num_simulation=500, num_workers=None)` â€” `num_simulation` is the playout budget *per worker*, so the effective budget is `num_simulation * num_workers` (default â‰ˆ 500 Ã— CPU count = 6000 on a 12-core box, ~4 s/move). `select_move(board)` reads `board.board`, runs playouts, and returns the most-visited root child move (or `None` only when the game is over / no legal moves). Each playout does UCT selection â†’ expand one child â†’ depth-capped (`ROLLOUT_DEPTH`) random rollout scored at the leaf by the configured **evaluator** (White-perspective in [0,1]) â†’ backpropagation that credits each node from the perspective of the side that moved into it. `Main.py`'s random-move fallback is now just a defensive net (unreachable unless MCTS gets a move-less position).
  - **Pluggable evaluator (the "better model" seam).** The one place a stronger model drops in is the **leaf evaluator**: a callable `board -> float` in [0,1] from White's perspective. `EnginePlayer(color, ..., evaluator="material")` selects it *by name*; the default `"material"` is `_material_evaluate` (material balance through a logistic). Register alternatives with `register_evaluator(name, fn)` (exported from `entity`) â€” **at import time**, because spawn workers re-import their entry module and each resolves the name against its own `EVALUATORS` registry. Evaluators cross the process boundary as a **name string, not a callable**, so a future neural net stays unpicklable-safe (build it once per worker via a pool `initializer`; don't ship a callable per task). The MCTS search, UCT, and backprop are unchanged â€” only the leaf scoring swaps. The stable external contract for any drop-in engine is just `select_move(board) -> Move|None` and `.color`.
  - **Root parallelization.** The MCTS phases (`_select`/`_expand`/`_rollout`/`_material_evaluate`/`_backpropagate`) and the tree driver `_search_visits(board, n, evaluator)` are **module-level** functions (colour-agnostic, no engine state), so worker processes can run them. `select_move` dispatches `num_workers` independent trees via a **reused** `multiprocessing.Pool` (lazy `self._pool`, `atexit`-cleaned): the module-level `_worker((fen, n, evaluator, chess960))` reseeds `random.seed()` (fresh OS entropy per task), rebuilds the root via `chess.Board(fen, chess960=chess960)` (the `chess960` flag is threaded through from `state.chess960` so Fischer-random castling is generated in workers â€” a Chess960 FEN parsed in standard mode silently drops castling), resolves the evaluator name locally, and returns `{move.uci(): visits}`; results are summed per move and the max wins. `num_simulation` is the budget **per worker**, so effective sims â‰ˆ `num_simulation * num_workers` in roughly one worker's wall-clock. `num_workers` defaults to `os.cpu_count()`; **`num_workers=1` forces the serial in-process path** (uses `state.copy()`, preserving move history), and any multiprocessing failure falls back to it too (sandboxes that forbid spawning still work). Because processes use spawn on Windows, worker code must stay import-safe â€” `Main.py` is `__main__`-guarded, and headless test scripts must be too. Note: workers rebuild the root from FEN, so rollout threefold-repetition detection differs negligibly from the serial `copy()` path (which keeps the full stack).
- **`SoundEffect`** (`entity/sound.py`) â€” owns loading and playing the five effect clips. `SoundEffect.load(sound_dir)` inits the mixer and loads `{name}.mp3` (missing files/headless mixer degrade to silence); `classify(board, move)` is a staticmethod naming the pre-push effect (Castle/Capture/Promote/Move); `play(name)` plays it. Behaviour matches the old inline logic exactly â€” Check overrides everything else (a capture-with-check plays only `Check.mp3`).
- **`BoardGenerator`** (`entity/board_generator.py`) â€” factory. Loads the 12 piece images **once up front** and builds the `SoundEffect` via `load_sounds()`, resolving `Images/` and `SoundEffect/` relative to the repo root via `__file__`. `generate(screen, fen=None, flipped=False, chess960=False)` returns a ready `Board` (passing `flipped` straight through to `Board`; `fen` is loaded as `chess.Board(fen, chess960=chess960)`). Images are keyed by `chess.Piece.symbol()` (e.g. `'P'`, `'k'`). `random_puzzle_fen(timeout=5)` is the **"Random" game-mode seam**: it fetches a real mid-game tactical puzzle from the Lichess puzzle API (`PUZZLE_URL = https://lichess.org/api/puzzle/next`, `urllib.request` â€” no new deps) and returns its FEN, or `None` on any failure (network/timeout, unexpected payload, illegal setup move). Reconstruction: Lichess gives a game `pgn` (space-separated SAN), an `initialPly`, and a `solution` (UCI moves); replay the pgn up to and including `initialPly`, then apply `solution[0]` â€” the opponent's auto-played "setup" move â€” to land on the position the solver actually faces (the tactic to solve), with that side to move. The result is a **standard** chess position, loaded with `chess960=False`. `random_chess960_fen()` (random Fischer/Chess960 start, `from_chess960_pos(randint(0,959))`, **must** load with `chess960=True`) is retained as the **offline fallback** for Random mode when a puzzle can't be fetched.

### Coordinate system / orientation

Board is 720Ã—720, `SQNUM=8`, `SQSIZE=90`. The board is oriented so **the human's chosen side sits at the bottom**, controlled by `Board.flipped` (set by `BoardGenerator.generate(..., flipped=...)`, which `main()` derives from the menu choice). Only the rank axis is mirrored â€” files stay a-to-h left-to-right in both orientations.

- **Not flipped** (`flipped=False`, default â€” human plays Black): **White at the top, Black at the bottom** (the original layout). **Row 0 (`y=0`) maps to rank index 0**: `square_at((0,0)) == chess.A1`, `square_at((0,630)) == chess.A8`; rank 1 sits at the top.
- **Flipped** (`flipped=True` â€” human plays White): the rank axis is mirrored so White sits at the bottom. `square_at((0,0)) == chess.A8`, `square_at((0,630)) == chess.A1`, `pixel_of(chess.A1) == (0,630)`.

`square_at`/`pixel_of` apply the flip (`rank = 7 - row` when flipped), and everything visual (pieces, move trace, selection, check highlight) routes through `pixel_of`, so a single flag flips the whole UI consistently. The round-trip invariant `square_at(pixel_of(sq)) == sq` holds in both orientations.

The entity implementation uses **standard chess files** (king on the e-file) â€” python-chess owns castling, en passant, and promotion. (The removed `Main_legacy.py` prototype swapped the king/queen files, putting the king on the d-file; the standard layout here is deliberate, not a bug.)

### Game loop (`Main.py`)

`main(max_frames=None, player_color=None, mode=None)` walks **two start menus**. First it resolves the human's side: when `player_color is None` it calls **`show_menu(screen)`** ("Choose Your Side" â€” "Play as White"/"Play as Black"). Then it resolves the game mode: when `mode is None` it calls **`show_mode_menu(screen)`** ("Choose Mode" â€” "Classic"/"Random"), returning `"classic"` or `"random"`. Both menus are drawn by a shared helper `_button_menu(screen, title, buttons)` (buttons stacked and centred; returns the clicked value or `None` on window-close). Random mode paints a "Loading puzzle..." frame (`_loading_screen`) then derives a `fen` from `BoardGenerator.random_puzzle_fen()` (a Lichess puzzle, `chess960=False`); if that returns `None` (offline/API failure) it falls back to `random_chess960_fen()` with `chess960=True` so the mode still works. Classic passes `fen=None`. The puzzle loads as-is â€” whichever side is to move plays first, human or engine. It then builds a `Board` via `BoardGenerator` (`flipped=(player_color == chess.WHITE)`), a `HumanPlayer(player_color)`, and an `EnginePlayer(not player_color)`. Each frame it draws, then: if it's the engine's turn (`board.board.turn == engine.color`) it plays the MCTS move (random fallback only if that returns `None`), after `ENGINE_DELAY` ms; it processes `QUIT` and left-click `MOUSEBUTTONDOWN` events. A click first checks `board.new_game_clicked(pos)` â€” when the game is over and the **"New Game"** button is hit, it rebuilds a fresh board via the `_new_board(screen, player_color, mode)` helper (same side/mode; a fresh puzzle in Random) â€” otherwise clicks route to the human player only on the human's turn. `board.board.is_game_over()` gates both sides. `_new_board` centralises the Random-mode puzzle fetch + Chess960 fallback + `flipped` derivation so both the initial build and every New Game restart share it; `max_frames` is a total budget across restarts. Because the engine is colour-agnostic (`_backpropagate` credits from the mover's perspective), it plays either side competently.

**Headless tests:** passing `max_frames` with `player_color=None` defaults the human to Black, and `mode=None` defaults to `"classic"` (both skip the blocking menus). Note that `main(max_frames=N, player_color=chess.WHITE)` plays **zero** moves within those frames â€” the engine is Black, it's the human's (White's) turn, and no click arrives â€” whereas the Black-human path lets the engine (White) move first.
